# AP-Univ-Planner 管理者向けガイド

この文書は、AP-Univ-Plannerの保守・更新・公開作業を行う管理者向けのメモです。  
一般利用者向けの説明は [`README.md`](../README.md) を参照してください。

## READMEとの役割分担

`README.md` は基本的に利用者向けの説明に限定します。  
講義データ更新、テスト、リリース前確認、GitHub Pages公開、管理者引き継ぎなど、一般利用者に不要な情報はこのファイルに集約します。

## ファイル構成

| ファイル / ディレクトリ | 内容 |

## ローカル確認

静的Webアプリのため、基本的にはローカルサーバーで開くだけで確認できます。

```bash
python3 -m http.server 8000
```

その後、ブラウザで `http://localhost:8000/` を開きます。

## 構文チェック

`script.js` を編集した場合は、少なくとも以下を実行してください。

```bash
node --check script.js
```

## 回帰テスト

カテゴリ判定やハイライト周辺を編集した場合は、以下を実行してください。

```bash
node tests/highlight_logic_test.js
```

主に以下を確認します。

- 所属専攻に基づくカテゴリ判定
- 同名科目のカテゴリ解決
- 通常枠・特殊枠のハイライト候補
- 集中講義枠での自関連・他関連の共存表示
- 旧形式保存データの互換読み込み
- カタログ拡張JSONの検証


## バージョン管理方針

AP-Univ-Plannerでは、**ツール本体のバージョン**と**講義カタログデータの更新日**を分けて管理します。

- ツール本体のバージョン: 画面上の `vX.Y.Z` やGitHub Releaseのタグで管理する
- 講義カタログデータの更新日: `data.js` 冒頭の `lastUpdated` で管理する

この2つを分けることで、UIやロジックの変更と、講義データの修正を区別しやすくします。

### ツール本体のバージョン

ツール本体のバージョンは、基本的に `vMAJOR.MINOR.PATCH` の形式で管理します。

| 種類 | 例 | 使う場面 |
|---|---|---|
| MAJOR | `v2.0.0` | 保存データ形式や利用手順が大きく変わり、互換性や使い方に大きな影響がある場合 |
| MINOR | `v1.1.0` | 新機能追加、UI改善、検証強化など、利用者に分かる機能追加・改善を行った場合 |
| PATCH | `v1.1.1` | 不具合修正、軽微なUI調整、文言修正、小規模な講義データ修正などの場合 |

現在公開済みのGitHub Releaseが `v1.0.0` の場合、その後に多数のUI改善・ロジック改善・検証強化を行った版は、通常は `v1.1.0` として扱うのが自然です。

保存JSON内の `version` は、アプリ本体のバージョンではなく、保存データ形式の内部バージョンです。GitHub Releaseの `vX.Y.Z` とは別物として扱ってください。

### 講義カタログデータの更新

講義カタログの修正を行った場合は、必ず `data.js` の `lastUpdated` を更新します。

講義データのみを修正した場合の扱いは、修正内容の重要度で判断します。

| 変更内容 | 推奨対応 |
|---|---|
| 誤字修正、軽微な表示修正 | `lastUpdated` の更新のみでも可 |
| 開講学期・曜日時限・開講なし等、履修判断に影響する修正 | `lastUpdated` 更新 + 必要に応じてPATCHリリース |
| 年度更新など、講義カタログ全体の大きな更新 | `lastUpdated` 更新 + GitHub Release作成を推奨 |

頻繁な小修正のたびに必ずGitHub Releaseを作る必要はありません。
ただし、履修判断に影響するデータ修正を公開した場合は、変更履歴としてGitHub Release本文やIssueに記録しておくと安全です。

### GitHub Releaseを作る基準

GitHub Releaseは、原則として**公開利用者に向けて区切りとして示したい変更**がある場合に作成します。

作成を推奨するケース:

- 画面上のツールバージョン `vX.Y.Z` を更新した
- 新機能や大きなUI改善を公開した
- 修了要件計算、カテゴリ判定、保存・読込など重要ロジックを変更した
- 年度更新など大きめの講義データ更新を行った
- 重要な不具合修正を公開した

作成しなくてもよいケース:

- READMEやドキュメントの軽微な修正のみ
- 表記ゆれやコメント修正のみ
- 開発用メモや内部管理ファイルの更新のみ

画面上のツールバージョンを更新する場合は、GitHub Releaseのタグと揃えてください。
例: 画面が `v1.1.0` なら、GitHub Releaseも `v1.1.0` を作成します。

### 推奨リリース手順

1. 次のバージョン番号を決める
2. 必要に応じて画面上のバージョン表示を更新する
3. 講義データを更新した場合は `data.js` の `lastUpdated` を更新する
4. `docs/RELEASE_CHECKLIST.md` に沿って確認する
5. GitHubへpushする
6. GitHub Pages反映後に公開URLで確認する
7. 必要に応じてGitHub Releaseを作成する

## 標準講義データの更新

標準講義データは `data.js` で管理しています。  
年度更新や公式情報の反映を行う場合は、`coreCourses` と `majorMasters` を編集してください。

データを更新した場合は、`data.js` 冒頭の `lastUpdated` も更新してください。  
画面上部の `Data: YYYY-MM-DD` 表示に反映されます。

### データ更新時の注意

- 公式シラバス、学生便覧、大学公開資料など、確認した出典をIssueやコミットメッセージに残してください。
- 開講なしの科目は、`name` と `schedule` のみでも構いません。
- 曜日時限がある科目は、`sem`, `day`, `period` を整合させてください。
- 集中講義は `isIntensive: true` を指定してください。
- 共同研究等の通常時限に入らない科目は `isOther: true` を指定してください。
- 同名科目が複数専攻に存在する場合、カテゴリ判定に影響するため、回帰テストを実行してください。

## サンプル履修データの更新

サンプル履修データは `examples/sample_plan.json` に置きます。  
現時点では、情報システム専攻の履修例1件のみを公開する方針です。

更新時は以下を確認してください。

- アプリの **保存(.json)** から出力した形式であること
- `myCourse` が `"1"` であること
- 氏名、学籍番号、メールアドレス、研究室名、個人的なメモなどの非公開情報が含まれていないこと
- アプリの **読込(.json)** から正常に読み込めること
- README、`docs/SAMPLE_PLAN.md`、`examples/README.md`、アプリ内リンクの説明とファイル名が一致していること

## カタログ拡張テンプレートの管理

配布用テンプレートは `templates/diff_template.json` に置きます。  
READMEやアプリ内リンクから参照されるため、配置場所を変更する場合はリンクも合わせて修正してください。

## Issueテンプレート

Issueテンプレートは、リポジトリ直下の `.github/ISSUE_TEMPLATE/` に配置します。  
`.git/` の中には置きません。

```text
.github/
└── ISSUE_TEMPLATE/
    ├── bug_report.yml
    ├── data_error_report.yml
    ├── feature_request.yml
    └── config.yml
```

## リリース前確認

公開前には `docs/RELEASE_CHECKLIST.md` を確認してください。

想定する流れは以下です。

1. コード修正が一段落する
2. `node --check script.js` を実行する
3. `node tests/highlight_logic_test.js` を実行する
4. ローカルサーバーでアプリを確認する
5. チェックリストに沿って、登録・削除・保存・読込・ハイライトなどを確認する
6. 問題がなければGitHubへpushする
7. GitHub Pages反映後に公開URLで最終確認する
8. 必要に応じてGitHub Release本文に確認結果を残す

`docs/RELEASE_CHECKLIST.md` はテンプレート兼手順書としてコミットし、チェックを入れた作業結果は原則コミットしません。確認結果を残す場合は、GitHub Release本文、Pull Request、Issueコメントなどを使ってください。

## GitHub Pages反映時の注意

GitHub Pagesは反映に少し時間がかかる場合があります。  
公開URLで確認する際は、ブラウザキャッシュの影響を避けるため、必要に応じてハードリロードしてください。

確認する主な項目は以下です。

- 画面が正常に初期化される
- `Data: YYYY-MM-DD` が意図した日付になっている
- 所属コース選択後に登録操作ができる
- 保存・読込が動作する
- サンプルJSONのリンクが有効である
- カタログ拡張テンプレートのリンクが有効である
- GitHub Issuesへのリンクが有効である

## 管理者引き継ぎ

引き継ぎ時は以下を確認してください。

- GitHubリポジトリの管理者権限
- GitHub Pagesの公開設定
- GitHub Releaseの運用状況
- Issue / Pull Requestの未対応項目
- `data.js` の最終更新日
- `docs/RELEASE_CHECKLIST.md`
- サンプル履修データの公開可否

長期的にメンテナンスを終了する場合は、リポジトリのアーカイブや後継リポジトリへの誘導も検討してください。
